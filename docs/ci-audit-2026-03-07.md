### CBOM-Root-Cause

`cdxgen --type cryptography` erkennt Go-stdlib-Crypto-Nutzung (`crypto/hmac`,
`crypto/sha256` etc.) nicht, weil stdlib-Pakete keine `go.mod`-Dependencies
sind und nicht via Source-Analysis aufgelöst werden. Das Artefakt war technisch
valides CycloneDX, aber inhaltlich leer — ein falsches Sicherheitsgefühl.

**Fix:** `--type cryptography` → `--type go`. Produziert ein echtes SBoM:

```
pkg:golang/github.com/danieljoos/wincred@v1.2.3
pkg:golang/golang.org/x/sys@v0.20.0
pkg:golang/golang.org/x/time@v0.14.0
```

Crypto-Algorithmen sind in `docs/security-concept.md` manuell dokumentiert.

## Workflow-Audit

### Befunde und Fixes

#### 1. `gosec` — falsch formatierter `#nosec`-Kommentar (`cmd/backup/main.go:27`)

**Problem:** gosec v2.24.7 meldet G104 auf `fs.Parse(os.Args[2:])`, obwohl
ein Suppressions-Kommentar vorhanden war. Ursache: `//nolint:errcheck // #nosec G104`
— gosec erkennt `#nosec` nicht, wenn es nach `//nolint:errcheck //` steht.

**Fix:** Kommentar zu `// #nosec G104 -- FlagSet uses ExitOnError; return value is unreachable` vereinfacht.

---

#### 2. `scorecard` — `Resource not accessible by integration`

**Problem:** `ossf/scorecard-action` benötigt für private Repos ein PAT mit
`repo` + `read:org` Scope. `GITHUB_TOKEN` kann Commits nicht via GraphQL
auflisten. Alle 6 Runs seit Einführung gescheitert.

Ursprünglicher Fix war `continue-on-error: true` — das maskierte den Fehler
(Workflow grün, obwohl Scan scheiterte).

**Endgültiger Fix:** Job läuft nur wenn `SCORECARD_ENABLED=true` als
Repository-Variable gesetzt ist, sonst `skipped`. Kein falsches Grün.

Aktivierung:
- `Settings → Variables → SCORECARD_ENABLED = true`
- `Settings → Secrets → SCORECARD_TOKEN = <PAT mit repo+read:org>`

---

#### 3. `commit-signature-verification` — warn-only ohne Key

**Problem:** Ohne `COMMIT_SIGNING_PUBLIC_KEY` lief der Job grün durch, auch
wenn alle Commits unsigniert waren. Das Script gibt nur `WARN: has no signature`
aus, schlägt aber nicht fehl — selbes Muster wie `scorecard`.

Zusätzlich: `github.event.inputs.*` wurden direkt in `run:`-Blöcke interpoliert
(potentielle Command-Injection).

**Fix:**
- Job läuft nur wenn `COMMIT_SIGNING_ENABLED=true` als Repository-Variable gesetzt
- `github.event.inputs.*` und `steps.range.outputs.*` werden via `env:` übergeben

Aktivierung:
- `Settings → Variables → COMMIT_SIGNING_ENABLED = true`
- `Settings → Secrets → COMMIT_SIGNING_PUBLIC_KEY = <GPG public key>`

---

#### 4. Unit-Test-Fehler in `internal/storage` — falsche Pfad-Traversal-Prüfung

**Problem:** `sanitizeName("export.pdf")` → `"export_pdf"` (Punkt wurde
ersetzt), dadurch scheiterte `TestWriter_WriteFile`.

Root Cause war der Regex `[^a-zA-Z0-9_\-]` der auch `.` ersetzte. Gleichzeitig
war die Containment-Prüfung `strings.HasPrefix(rel, "..")` fehlerhaft: sie
flaggte sanitisierte Namen wie `".._.._etc_passwd.json"` als Path-Traversal
(False Positive), da der String zufällig mit `".."` beginnt — ohne zu prüfen,
ob es sich um eine echte Pfad-Komponente handelt.

**Fix:**
- Regex erweitert: `[^a-zA-Z0-9_\-\.]` — Punkte für Dateiendungen erlaubt
- Neue Funktion `isOutsideDir(rel)`: prüft `rel == ".."` oder
  `strings.HasPrefix(rel, "../")` — pfadkomponenten-bewusst statt
  reine String-Prefix-Prüfung
- Test-Erwartungswert angepasst: `"______etc_passwd.json"` → `".._.._etc_passwd.json"`
- `TestWriteJSONStaysWithinBackupDir` verwendet dieselbe pfadkomponenten-bewusste Prüfung
